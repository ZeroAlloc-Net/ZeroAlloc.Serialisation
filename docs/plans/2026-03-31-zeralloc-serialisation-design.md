# ZeroAlloc.Serialisation — Design

**Date:** 2026-03-31  
**Status:** Approved

## Goal

A shared `IBufferWriter<byte>`-based serialisation library consumed by both `ZeroAlloc.Rest` and the upcoming `ZeroAlloc.EventSourcing`. Provides typed, AOT-safe `ISerializer<T>` primitives with source-generated implementations per annotated type.

## Why

- Single serializer implementation per format (MemoryPack / MessagePack / STJ) — no duplication across Rest and EventSourcing.
- `IBufferWriter<byte>` eliminates the intermediate `byte[]` allocation present in the current `MemoryPackRestSerializer`.
- Source generator produces closed-generic implementations free of `[RequiresDynamicCode]` / `[RequiresUnreferencedCode]`.
- EventSourcing uses `ISerializer<T>` directly; REST bridges via a thin adapter — no forcing EventSourcing to take a REST dependency.

## Core Interface & Attribute

```csharp
// ZeroAlloc.Serialisation
public interface ISerializer<T>
{
    void Serialize(IBufferWriter<byte> writer, T value);
    T? Deserialize(ReadOnlySpan<byte> buffer);
}

public enum SerializationFormat { MemoryPack, MessagePack, SystemTextJson }

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Struct)]
public sealed class ZeroAllocSerializableAttribute(SerializationFormat format) : Attribute
{
    public SerializationFormat Format { get; } = format;
}
```

## Source Generator

**Input:** Any type decorated with `[ZeroAllocSerializable(SerializationFormat.X)]`.

**Output per type:**

1. A sealed `{TypeName}Serializer : ISerializer<{TypeName}>` class that calls the backend's static API with a closed generic — no `RequiresDynamicCode`.
2. A DI registration extension method.

**Example** (MemoryPack backend):

```csharp
// <auto-generated>
internal sealed class PersonEventSerializer : ISerializer<PersonEvent>
{
    public void Serialize(IBufferWriter<byte> writer, PersonEvent value)
        => MemoryPackSerializer.Serialize(writer, value);

    public PersonEvent? Deserialize(ReadOnlySpan<byte> buffer)
        => MemoryPackSerializer.Deserialize<PersonEvent>(buffer);
}

// DI extension:
public static IServiceCollection AddPersonEventSerializer(this IServiceCollection services)
    => services.AddSingleton<ISerializer<PersonEvent>, PersonEventSerializer>();
```

AOT safety: `T` is closed at generation time. MemoryPack's own generator has already emitted the formatter for `PersonEvent`, so no dynamic code is required.

## Backend Packages

Each backend package provides a non-generated base class for ad-hoc / non-AOT use, and the attribute that triggers the generator:

| Package | Base class | Constraint |
|---|---|---|
| `ZeroAlloc.Serialisation.MemoryPack` | `MemoryPackSerializer<T>` | `where T : IMemoryPackable<T>` |
| `ZeroAlloc.Serialisation.MessagePack` | `MessagePackSerializer<T>` | none (uses options) |
| `ZeroAlloc.Serialisation.SystemTextJson` | `SystemTextJsonSerializer<T>` | requires `JsonTypeInfo<T>` |

## REST Integration

`ZeroAlloc.Rest` takes a new optional dependency on `ZeroAlloc.Serialisation`. Two changes:

1. `RestSerializerAdapter<T>` — bridges `ISerializer<T>` to `IRestSerializer` (rents `ArrayBufferWriter<byte>`, calls `ISerializer<T>.Serialize`, writes buffer to stream).
2. `ZeroAlloc.Rest.MemoryPack` internally fixed to write via `ArrayBufferWriter<byte>` instead of allocating a `byte[]`.

`IRestSerializer` is unchanged — no breaking change for existing users.

## EventSourcing (future)

References `ZeroAlloc.Serialisation` only. Decorates event types with `[ZeroAllocSerializable]`, gets generated `ISerializer<TEvent>` implementations + DI wiring. No dependency on `ZeroAlloc.Rest`.

## Testing

- Unit tests per backend: round-trip `Serialize` → `Deserialize` for primitive and complex types.
- Generator tests: verify emitted code compiles and produces correct output for each `SerializationFormat`.
- Benchmark: `ISerializer<T>` path vs current `IRestSerializer` Stream path (allocation delta).
