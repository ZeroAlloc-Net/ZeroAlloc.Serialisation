# Bebop C# API research (V3 viability)

Date: 2026-06-10
Status: research-only

## TL;DR

The evidence supports **(b) — sharpen the backlog deferral with structural rationale**, leaning towards **(c)**. Bebop's C# experience is fundamentally **schema-first**: the user writes `.bop` files, `bebopc` emits generated record classes at build time, and there is no public `IBebopFormatter<T>` / per-type converter interface analogous to MessagePack's `IMessagePackFormatter<T>` or STJ's `JsonConverter<T>` that user types can implement to opt into custom serialization. The "same shape as MessagePack/MemoryPack" claim in the backlog is incorrect — Bebop has no attribute-driven hook for an existing user struct, so a `[ValueObject] OrderId` cannot be plugged into Bebop without redefining `OrderId` inside a `.bop` schema (which destroys the typed-ID semantics that motivate ZA.Serialisation in the first place).

However, the `BebopWriter`/`BebopReader` low-level API (ref structs with `WriteInt32`/`ReadInt32` primitives etc.) is public and AOT-friendly. A *constrained* implementation is possible if we accept that VOs would only work when embedded inside an existing user-authored `[BebopRecord]` parent via direct primitive read/write — never as a standalone Bebop message. That is a meaningfully smaller use case than STJ/MessagePack/MemoryPack offer, and is not "same shape."

## Q1 — C# runtime library

- **Package:** [`bebop`](https://www.nuget.org/packages/bebop/) on NuGet (owner: AndrewMD5)
- **Latest stable:** **3.2.3** (published 2025-09-16)
- **Target frameworks:** net8.0, net9.0
- **AOT:** Bebop v3 explicitly advertises full NativeAOT support — confirmed in both the GitHub README and v3 release notes. Bebop has *zero runtime dependencies*, which is consistent with AOT-friendliness.
- **Companion package:** [`bebop-tools`](https://www.nuget.org/packages/bebop-tools/) (3.2.1) — MSBuild integration that runs the `bebopc` compiler on `.bop` files as part of the build. The package ships the compiler binary; adopters do **not** need to install `bebopc` separately as a CLI. License Apache-2.0.

Adoption is modest: ~56k total downloads on the runtime package, ~7.5k for the current version. This is in a different volume class than `MessagePack` (~hundreds of millions) or `MemoryPack` (~tens of millions).

## Q2 — Reader/writer API

Both `BebopReader` and `BebopWriter` are **`public ref struct`**s — the exact same shape as `MessagePackReader`/`MessagePackWriter`. This part of the architecture is friendly to a generator-emitted formatter pattern.

### `BebopWriter` (excerpt)

```csharp
public ref struct BebopWriter
{
    public static BebopWriter Create();
    public static BebopWriter Create(int initialCapacity);
    public static BebopWriter Create(byte[] buffer);
    public static BebopWriter From(Span<byte> buffer);

    public void WriteByte(bool value);
    public void WriteByte(byte value);
    public void WriteInt16(short value);
    public void WriteUInt16(ushort value);
    public void WriteInt32(int value);
    public void WriteUInt32(uint value);
    public void WriteInt64(long value);
    public void WriteUInt64(ulong value);
    public void WriteFloat32(float value);
    public void WriteFloat64(double value);
    public void WriteDate(DateTime date);
    public void WriteGuid(Guid guid);
    public void WriteString(string value);
    public void WriteBytes(byte[] value);

    public int  ReserveRecordLength();
    public void FillRecordLength(int position, uint messageLength);

    public ReadOnlySpan<byte> Slice();
    public byte[]             ToArray();
}
```

### `BebopReader` (excerpt)

```csharp
public ref struct BebopReader
{
    public byte    ReadByte();
    public ushort  ReadUInt16();
    public short   ReadInt16();
    public uint    ReadUInt32();
    public int     ReadInt32();
    public ulong   ReadUInt64();
    public long    ReadInt64();
    public float   ReadFloat32();
    public double  ReadFloat64();
    public DateTime ReadDate();
    public string   ReadString();
    public Guid     ReadGuid();
    public uint     ReadRecordLength();
}
```

Every read method is annotated with `[MethodImpl(BebopConstants.HotPath)]`. Naming convention is `Int32`/`Float32` (Bebop's schema language uses those names) rather than `.NET`-idiomatic `Int`/`Single` — this is a cosmetic difference vs MessagePack, not a structural one.

## Q3 — Custom converter hook  (load-bearing)

**This is where the analogy to the other backends breaks down.**

Bebop's runtime model:

- `BebopRecord` is a **class** in `Runtime/C#/Runtime/BebopRecord.cs`, generated by `bebopc`. It is *not* a user-implementable interface. There is no `IBebopFormatter<T>` analog. There is no `IBebopConverter<T>` analog.
- `BebopRecord<T>` is a **sealed concrete** wrapper produced by the generator, where `T : BaseBebopRecord`. Adopters do not subclass it.
- The discovery mechanism is `BebopMirror` — a reflection-based registry built at process start by scanning *all loaded assemblies* for types decorated with `BebopRecordAttribute`. Users can attach handler methods via `[BindRecord(typeof(BebopRecord<ChatMessage>))]` on a class marked `[RecordHandler]`, but the handlers are for **dispatch on decode**, not for *defining how a type encodes/decodes itself*.
- The closest attribute to MessagePack's `[MessagePackFormatter(typeof(...))]` is `BebopRecordAttribute`, but it's a **marker emitted by the code generator**, not a hook for the user to redirect serialization to a custom type.

In other words: Bebop's design assumption is "the schema compiler owns the encode/decode methods for every record." There is no public extension point that says "for type `X`, call *this* encode method instead of the generator's." The only place where a custom type can plug in is inside a user-authored `.bop` schema, by referencing another user-defined record / enum — and those references *also* have to be Bebop-generated types.

### Could we still emit something useful?

Yes, in principle, but in a *narrower* way than the other backends:

- We can emit `public void WriteTo(ref BebopWriter w)` and `public static OrderId ReadFrom(ref BebopReader r)` instance/static members on the partial-struct extension. These are just plain methods using public primitives (`WriteInt32`/`ReadInt32`). No registration, no attribute, no interface implementation needed.
- The adopter would then need to **hand-call** `orderId.WriteTo(ref writer)` inside the `Encode` method of *their* `[BebopRecord]` — i.e. inside generator-emitted code that they don't normally edit, because `bebopc` owns those files.

That last constraint is the killer. With STJ/MessagePack/MemoryPack, the runtime *discovers* the converter/formatter at dispatch time (`JsonSerializerOptions`, `IFormatterResolver`, module initializer) and the adopter never edits the encode site. With Bebop, there is no discovery path: the adopter would either have to (a) hand-edit generated `.cs` files (fragile, breaks on regen), or (b) redefine `OrderId` as a Bebop struct inside `.bop` (loses VO semantics).

## Q4 — Wire format

Bebop has two record kinds with different framing:

- **`struct`** — fields concatenated in order, no header, no length prefix, no field index. This is the natural shape for embedding "primitive-equivalent" VOs.
- **`message`** — length prefix (`uint32`) **plus** per-field index tags. Variable-length, forward-compatible. The writer reserves the length slot via `ReserveRecordLength()` and fills it via `FillRecordLength(...)`.

**Bare primitives:** `BebopReader.ReadInt32()` and `BebopWriter.WriteInt32(...)` do work standalone on a buffer that contains only a 4-byte little-endian int — the reader and writer are positional and don't require any framing. So technically a VO wrapping `int` could round-trip as 4 raw bytes. But: this is *not* a valid Bebop "message" on the wire — there's no opcode, no length, nothing for `BebopMirror` to dispatch on. It's just bytes.

**Embedded in a parent record:** If a VO is referenced from a parent `[BebopRecord] struct Foo { ... }`, the natural encoding is to inline its single primitive — i.e. the VO is wire-format-identical to its underlying primitive. That is *the* sane use case for our pattern. But the parent record's generated encode method has no extension point that says "call `OrderId.WriteTo(ref w)` here" — `bebopc` would just emit `_writer.WriteUInt64(_field.RawValue)` if the field were `ulong`, but if the field is `OrderId` it has no idea what to do because `OrderId` isn't in the `.bop` schema.

## Q5 — Schema-first vs attribute-first

**Bebop is schema-first by design, not by accident.** The entire workflow is:

1. Write `.bop` schema (e.g. `struct Person { string name; uint32 age; }`)
2. Add `<Bebop Include="**/*.bop" />` to the `.csproj`
3. `bebop-tools` runs `bebopc` during build and emits `.cs` files
4. Adopter consumes the generated `Person` type via `Person.Encode(...)` / `Person.Decode(...)`

There is **no attribute-driven mode** where an adopter decorates an existing C# type with `[BebopRecord]` and Bebop "adopts" it. The attribute exists but is **emitted by the generator onto generated code**, not authored by humans on hand-written types.

### Concrete consequence for `[ValueObject] struct OrderId`

The adopter has a couple of bad options and no good ones:

- **Option A (redefine in `.bop`):** Add `struct OrderId { uint64 value; }` to a `.bop` file. Now there is a generated `OrderId` class *and* a hand-written `[ValueObject]` struct — name collision, two types, and the generated one is a class without the typed-ID semantics, equality, etc. that motivated the VO.
- **Option B (just use the primitive):** Drop the VO and use `ulong` in the `.bop`. Loses everything ZA.Serialisation is trying to provide.
- **Option C (emit `WriteTo`/`ReadFrom` and hand-wire):** What our generator can do — but as Q3 noted, the adopter has nowhere to call these from, because they don't author the encode site.

None of these are the smooth "decorate your VO and it just works in your existing serializer" experience that we deliver for STJ/MessagePack/MemoryPack.

## Q6 — Adopter use case

Bebop positioning:

- Self-described as a **modern Protobuf/gRPC alternative** focused on schema-first binary serialization.
- Has a companion RPC framework called **Tempo** that builds on Bebop.
- Use cases mentioned: real-time APIs, browsers, serverless, bare metal, IPC, "performance-critical data".
- Cross-language: C#, TypeScript, Rust, C++.

This is **a fundamentally different adopter profile** from MessagePack/MemoryPack:

- MessagePack/MemoryPack adopters typically have a .NET-first codebase and use serialization for caching, queues, gRPC contracts, etc. They reach for attribute-driven decoration of existing types.
- Bebop adopters are typically polyglot teams who deliberately chose a schema-first IDL because they *want* the schema to be the source of truth across multiple languages. In that worldview, a VO is an implementation detail of one language and has no place in the schema.

There is **no evidence** any current ZA adopter uses Bebop. The backlog framing ("scales cleanly — same shape as the MessagePack/MemoryPack additions") appears to have been written without checking the Bebop API surface — it's a category error.

## Recommendation

**(b) — sharpen the backlog deferral with structural rationale, leaning to (c).**

The "Bebop scales cleanly" claim in the V3 backlog is incorrect and should be replaced with one of two structural findings:

1. **Defer indefinitely** with the note: *Bebop is schema-first by design. It has no per-type formatter interface (no `IBebopFormatter<T>` analog) and no attribute-driven hook on user-authored types. Plugging a `[ValueObject]` into Bebop requires either redefining the type in `.bop` (loses VO semantics) or hand-editing generator output (fragile). Until Bebop ships an opt-in extensibility API for user-defined "external" types, this backend cannot match the fidelity of the STJ/MessagePack/MemoryPack integrations.*

2. **Drop V3** if architectural completeness was the only motivation. The remaining option — emitting `WriteTo`/`ReadFrom` primitive-passthrough methods — has no clean dispatch site on the adopter end, so it would solve a non-problem.

### Optional reduced-scope sketch (only if we still want *something*)

If V3 ships anyway under a very narrow scope, the realistic emission would be:

```csharp
// Generated extension on a [ValueObject] struct OrderId backed by ulong
partial struct OrderId
{
    public void WriteTo(ref global::Bebop.Runtime.BebopWriter w)
        => w.WriteUInt64(_value);

    public static OrderId ReadFrom(ref global::Bebop.Runtime.BebopReader r)
        => new OrderId(r.ReadUInt64());
}
```

with documentation explicitly stating that the adopter has to hand-call these inside their own (non-Bebop-generated) encode/decode code. No attribute. No `BebopMirror` registration. No `[BebopFormatter]`. It's a much weaker promise than the other backends offer, and the docs cost may outweigh the value.

My recommendation is **(b) with the structural rationale above written into the backlog**, and to revisit only if Bebop itself adds a user-extensibility API in a future release.
